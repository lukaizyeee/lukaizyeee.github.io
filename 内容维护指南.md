# 网站内容维护指南

这份文件教你以后**不需要找 Claude**，自己就能改网站上的文字内容：写博客、加项目、更新「此刻」页、改简历。

核心心法只有一句话：**网站的"内容"和"设计"是分开的**。内容全部是 `src/content/` 目录下的 Markdown 文件（外加 About 页是两个 `.astro` 文件），你只需要编辑这些文件里的文字，改完 `git push`，两三分钟后网站自动更新。你不需要碰任何 `.astro` 组件、CSS 或配置文件。

---

## 0. 最快上手：三步走

1. 在 `src/content/blog/`（或 `projects/`、`now/`）里新建或编辑一个 `.md` 文件。
2. 本地看一眼效果（可选但推荐）：`astro dev --background`，然后打开 http://localhost:4321。
3. 确认没问题后：

```bash
git add -A
git commit -m "内容更新：xxx"
git push
```

push 到 `main` 分支后，GitHub Actions 会自动构建并部署，几分钟后 https://aizyeee.github.io 就是最新内容。不需要手动运行 build 或者手动上传任何东西。

---

## 1. 本地预览网站

```bash
astro dev --background   # 后台启动开发服务器
astro dev status         # 看是否在跑、跑在哪个端口
astro dev logs           # 看日志（报错通常在这里）
astro dev stop           # 停止
```

启动后打开 http://localhost:4321 即可看到实时预览（改文件会自动刷新）。如果你只是改改文字、不追求可视化确认，也可以跳过这一步，直接改完就 push——GitHub Actions 构建时如果 Markdown 格式写错了，会直接报错并且不会部署（不会把错误内容发布上线），所以本地不预览也不用太担心"发布出错内容"。

## 2. 网站内容都在哪：四个地方

```
src/content/blog/          博客文章（中英文混放，一篇文章只需要写一种语言）
src/content/projects/en/   项目介绍 · 英文
src/content/projects/zh/   项目介绍 · 中文（文件名要和 en/ 对应）
src/content/now/           「此刻」页（en.md / zh.md 各一份，只有这两个文件）
src/pages/about.astro      「关于」页 · 英文（直接改文字，不是 Markdown）
src/pages/zh/about.astro   「关于」页 · 中文
```

下面逐个讲怎么改。

### 2.1 博客文章 `src/content/blog/`

每篇文章一个 `.md` 文件，文件名会变成网址（比如 `my-notes.md` → `你的域名/blog/my-notes`）。**文件名请用英文/拼音+短横线，不要用中文或空格**，避免网址出现乱码转义。

文件开头是 frontmatter（`---` 包裹的部分），字段说明：

| 字段 | 说明 | 示例 |
|---|---|---|
| `title` | 标题 | `"聊聊这次的项目重构"` |
| `date` | 发布日期，`YYYY-MM-DD` | `2026-07-04` |
| `lang` | 这篇是中文还是英文，只能填 `en` 或 `zh` | `zh` |
| `tags` | 标签，数组格式 | `[ai, 随笔]` |
| `description` | 一句话摘要，会显示在列表页和 RSS | `"这周做的一些实验记录"` |
| `draft` | 选填，`true` 的话不会发布（草稿） | `true` |

模板（直接复制新建文件即可）：

```markdown
---
title: "标题写在这里"
date: 2026-07-04
lang: zh
tags: [ai, 随笔]
description: "一句话摘要"
---

正文从这里开始，正常 Markdown 语法：**加粗**、`代码`、列表、二级标题 `##` 等等都支持。
```

**博客不需要中英对照**：一篇文章选一种语言写就行，`lang` 字段会让它自动出现在对应语言的博客列表和 RSS 里。想写草稿先占位、以后再发，就设 `draft: true`，网站不会显示它，去掉这行或改成 `false` 再发布。

新建文件后**不需要**去别处登记——博客列表页、首页"最近文章"、RSS 订阅都是自动从这个文件夹里读取并按日期排序的。

### 2.2 项目 `src/content/projects/en/` 和 `.../zh/`

项目页是双语成对的：`en/` 和 `zh/` 各放一份，**文件名必须完全一致**（比如 `en/lighter.md` 对应 `zh/lighter.md`），这样网站顶部的中英文切换按钮才能跳转到对应的中/英文版本；如果只写了一边、忘了写另一边，切换按钮会退回到项目列表页（不会报错，但体验打折扣，所以尽量两边一起写全）。

字段说明：

| 字段 | 说明 | 示例 |
|---|---|---|
| `title` | 项目名 | `"合同智能审查系统"` |
| `period` | 时间段 | `"2025"` |
| `role` | 你的角色 | `"独立开发者"` |
| `summary` | 一句话摘要，显示在卡片上 | `"基于 RAG 的合同风险识别工具"` |
| `tech` | 技术栈标签，数组 | `[Python, RAG, FastAPI]` |
| `metrics` | 量化成果标签，数组，选填 | `["900+ 测试用例", "准确率 92%"]` |
| `featured` | 是否是"精选项目"（会出现在首页） | `true` |
| `order` | 排序用，数字越小越靠前 | `1` |
| `cover` | 封面图路径，选填（目前页面还没接入展示，先留空即可） | — |

模板：

```markdown
---
title: "项目名"
period: "2025"
role: "独立开发者"
summary: "一句话说清楚这个项目做了什么"
tech: [Python, RAG]
metrics: ["一个量化成果", "另一个量化成果"]
featured: true
order: 1
---

## 背景与动机
正文……

## 系统架构
正文……

## 关键设计决策
正文……

## 数据与成果
正文……

## 反思与下一步
正文……
```

（二级标题可以按自己的项目情况增删，不是强制的固定模板，只是现有两篇的写法，保持风格一致即可。）

关于 `featured` 和 `order` 再多说一句：`featured: true` 的项目会出现在首页"精选项目"和项目列表页的置顶分组（带摘要卡片）；`featured: false`（或不写）的项目只会出现在项目列表页下方的"Other projects"里（只显示标题，没有摘要）。首页会展示**所有** `featured: true` 的项目，不会自动截断，所以如果精选项目定多了，首页会变长——建议精选项目控制在 3〜5 个左右。`order` 数字额外还会决定卡片配色是米白还是薰衣草紫的交替（纯视觉效果，不用管，正常填排序数字就行）。

### 2.3 「此刻」页 `src/content/now/`

只有两个文件：`en.md`（英文）和 `zh.md`（中文），不需要新建、只需要编辑这两个文件。字段只有一个 `updated`（更新日期），每次改动记得把这个日期也改成当天。

```markdown
---
updated: 2026-07-04
---

**在读** —— 正在看的书/论文。**在做** —— 最近在捣鼓的东西。**在想** —— 最近的一些想法。
```

正文格式随意，不强制这三段式，但目前是这个风格，建议保持简短（这一页的定位就是"一句话近况"，不是长文）。

### 2.4 「关于」页（不在 `src/content/`，是普通页面文件）

`src/pages/about.astro`（英文）和 `src/pages/zh/about.astro`（中文）。这两个文件看起来是代码文件，但你只需要改**引号里的文字部分**（简介、技能标签、邮箱、链接），不用动 `<style>` 块和标签结构，改文字是完全安全的。比如简介这一行：

```astro
<p>Placeholder bio — replaced in content phase. ECE undergrad at Shanghai Jiao Tong University (2024–2028), working toward AI systems and OS × AI research.</p>
```

把引号里的文字换成你自己的简介即可，其他 `<h2>Skills</h2>`、邮箱链接、GitHub 链接同理直接改文字/链接地址。如果不确定某处该不该动，或者想加新的段落/新的一块内容（比如加一个"经历"板块），这种涉及结构调整的可以来找我帮你改，纯文字替换你自己动手完全没问题。

## 3. 图片和简历文件

静态文件（图片、PDF 等）放在 `public/` 目录下，网站里引用时路径不带 `public` 前缀，比如 `public/photo.jpg` 在页面里写 `/photo.jpg`。

简历文件是 `public/resume.pdf`，**更新简历只需要用新文件覆盖替换这一个文件，文件名保持 `resume.pdf` 不变**（About 页的下载按钮链接是写死的 `/resume.pdf`）。如果换了别的文件名，需要同步去 `about.astro` 两个文件里把链接改掉，这种情况可以找我处理。

## 4. 发布上线

改完内容后：

```bash
git add -A
git commit -m "描述这次改了什么"
git push
```

推送到 `main` 分支会自动触发 GitHub Actions（可以在仓库的 Actions 标签页看进度，通常 2〜3 分钟跑完），跑完后网站自动更新，不需要任何手动部署步骤。

**如果 Actions 显示部署失败**："Deployment failed, try again later" 这类报错大概率是 GitHub Pages 服务端的偶发抽风，不是你内容写错了——去 Actions 页面对那次失败的 run 点 "Re-run jobs" 通常就成功了。如果是构建（build）那一步报错、不是部署（deploy）那一步，那才是内容格式的问题，看下一节排查。

---

## 5. 常见报错排查

内容格式有严格校验（写错了 GitHub Actions 会直接报错、不会发布出错的内容，这是有意为之的保护机制），常见错误：

- **日期格式不对**：`date` / `updated` 必须是 `YYYY-MM-DD`，不要写"2026年7月4日"这种。
- **数组字段忘记加括号**：`tags`、`tech`、`metrics` 必须是 `[a, b, c]` 这种数组格式，不能直接写 `tags: 随笔`。
- **`lang` 字段拼错**：博客的 `lang` 只能是 `en` 或 `zh`，其他值会报错。
- **frontmatter 的 `---` 缺失或缩进错乱**：一定是开头一行 `---`、结尾一行 `---`，中间是字段。
- **必填字段漏写**：博客必填 `title/date/lang/description`；项目必填 `title/period/role/summary/tech`。

报错信息会出现在 GitHub Actions 的构建日志里，通常会明确指出是哪个文件、哪个字段的问题（比如 "Invalid date" 或 "Expected array, received string"）。如果看不懂报错内容，把报错信息发给我就行。

---

## 6. 不建议自己碰的部分

以下这些是"设计系统"和"基建"，改动会影响视觉效果或者网站的正常运转，建议改动前先跟我讨论：

- `src/components/*.astro` —— 所有组件（按钮、卡片、导航栏等的样式和动效）
- `src/styles/tokens.css` —— 全局色彩、字体变量
- `src/layouts/BaseLayout.astro` —— 页面整体骨架
- `astro.config.mjs`、`.github/workflows/`、`scripts/check-dist.sh` —— 网站配置和部署流程

如果以后想加新的页面类型、调整版式、换配色方案，这些是"设计变更"，找我来做就行，走一遍讨论再实现的正常流程。**日常的文字更新（写文章、加项目、改简介）完全不涉及这些文件**，放心大胆地改 `src/content/` 和两个 `about.astro` 里的文字。
